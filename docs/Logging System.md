# 🍇 Grape Engine Logging System

## Software Engineering Document

---

# 1. Introduction

The **Grape Engine Logging System** is the event logging subsystem of the Grape Engine.

It is designed to be:

* ⚡ **High-performance**
* 🧱 **Modular**
* 🔌 **Pluggable**
* 🧠 **Predictable**
* 🔄 **Future-proof (async-ready)**

The system allows any part of the engine to generate logs **without knowing how or where they will be consumed**.

### Design Philosophy

The logger follows the core principles of the Grape Engine:

* **Modularity** → Backends are fully decoupled
* **Performance** → No dynamic allocation in hot path
* **Self-contained data** → Events do not depend on external memory
* **Low coupling** → Systems communicate through structured data
* **Evolution-ready** → Designed to support async, ring buffers, and event systems

---

# 2. System Goal

The logger solves a fundamental problem:

> How to log information without coupling producers to output systems?

The solution is a 3-layer architecture:

1. **Public API** → Macros and functions used by the engine
2. **Logger Core** → Creates events and dispatches them
3. **Backends** → Consume events (console, file, network, etc.)

---

# 4. Core Data Structures

## 4.1 Log Levels

```c
enum GrapeLogLevel {
    GRAPE_LEVEL_DEBUG,
    GRAPE_LEVEL_INFO,
    GRAPE_LEVEL_WARNING,
    GRAPE_LEVEL_ERROR,
    GRAPE_LEVEL_FATAL,
};
```

Defines severity of events.

---

## 4.2 Log Categories

```c
typedef uint32_t GrapeLogCategory;
```

Bitmask-based system:

```c
#define GRAPE_LOG_RENDER  (1u << 0)
#define GRAPE_LOG_AUDIO   (1u << 1)
#define GRAPE_LOG_PHYSICS (1u << 2)
#define GRAPE_LOG_ECS     (1u << 3)
#define GRAPE_LOG_IO      (1u << 4)
```

Allows fast filtering:

```c
GRAPE_LOG_RENDER | GRAPE_LOG_ECS
```

---

## 4.3 Log Event (Core Structure)

```c
typedef struct GrapeLogEvent {
    uint64_t timestamp;
    uint64_t sequence;
    const char* file;
    const char* function;
    uint32_t line;
    uint32_t category;
    uint8_t level;
    char message[GRAPE_LOG_MESSAGE_MAX];
    void* user_data;
} GrapeLogEvent;
```

### Design Decision: Self-Contained Event

The `message` is stored **inside the struct**.

### Why?

* ✅ Safe lifetime (no dangling pointers)
* ✅ Thread-safe data transfer
* ✅ Async-ready
* ✅ No dependency on stack memory
* ✅ Deterministic memory layout

### 🔢 Sequence Field

The `sequence` field represents a strictly increasing counter generated by the logger core.

### Why it exists:
- Guarantees total ordering of log events
- Resolves same-timestamp ambiguity
- Enables deterministic debugging and replay
- Supports future async and multithreaded logging systems

### Important property:

>`sequence` is guaranteed to be unique and strictly increasing per runtime execution.

---

This is critical for future features like:

* async logging
* ring buffers
* event systems
* debugging determinism

---

## 4.4 Backend

```c
typedef struct Backend {
    void (*write)(const GrapeLogEvent* event, void* user_data);
    void* user_data;
} Backend;
```

Each backend is:

* independent
* replaceable
* fully decoupled

### ⚠️ Backend Contract

Backends MUST treat GrapeLogEvent as temporary data.

The event is only valid during the execution of write().

Backends MUST NOT store pointers to the event or its internal fields.
If persistence is required, data must be copied inside the callback.

---

## 4.5 Logger State

```c
typedef struct GrapeLoggerState {
    enum GrapeLogLevel level;
    Backend backends[GRAPE_LOGGER_MAX_BACKENDS];
    uint32_t backend_count;
    GrapeLogCategory category_mask;
} GrapeLoggerState;
```

Stored as:

```c
static GrapeLoggerState g_logger;
```

Encapsulated inside the logger core.

# 9. Example Backend

```c
void console_write(const GrapeLogEvent* event, void* user_data) {
    // Safe usage: only reads during callback
    printf("[%llu] %s:%u %s(): %s\n",
        (unsigned long long)event->timestamp,
        event->file,
        event->line,
        event->function,
        event->message);
}
```

# 10. Key Design Decisions

| Decision              | Reason                 |
| --------------------- | ---------------------- |
| Self-contained event  | Safe, async-ready      |
| No dynamic allocation | Performance            |
| Fixed-size message    | Predictability         |
| Backend abstraction   | Modular system         |
| Early filtering       | Avoid unnecessary work |
| Static global state   | Simplicity             |
| Sequence counter      | Ensures deterministic ordering of events             |

# 12. Current Limitations

### 12.1 Fixed message size

Limited by `GRAPE_LOG_MESSAGE_MAX`.

---

### 12.2 Synchronous dispatch

Logging is currently synchronous.

---

### 12.3 Fixed backend count

Static array (16 backends).

---

### 12.4 Not thread-safe

Current implementation assumes single-thread usage.
